# Minerva 改版規格：從「聊天教練」到「AI 日記」

> 2026-07-22 定案。這份是實作的唯一對照，做的時候有分歧以這份為準。
> 原則：**簡化、收斂、有邊界**。沒用到的舊 code 直接刪，不留註解式的墓碑。

---

## 為什麼改

現在的版本是「講一句、回一句」的即時對話。問題不是做得不好，是**看不出系統做了什麼** —— 畫面上跟 ChatGPT 沒兩樣，pgvector、事實抽取、profile 全是隱形的。

改成日記之後，同一套技術從隱形變成賣點：一天一篇 → 每篇跑出結構化結果 → 累積成能量趨勢。打開就看到**只有「有記憶的系統」才生得出來的畫面**。

---

## 一、核心規則（產品的邊界）

| 規則 | 內容 |
|---|---|
| 一天一篇 | 一個 uid + 一個台灣日期 = 最多一篇日記。DB 層 unique 索引擋死 |
| 過去就過去了 | 只有「今天」能寫、能改。過去的日期唯讀，不能補寫、不能編輯 |
| 寫日記不限次 | 一天會分好幾次補記，收費等於懲罰你跟上自己的日子；存文字也不花錢 |
| 分析一天 3 次 | 這才是花 LLM 的動作，也是「一天該收尾」該管的地方 |
| 手動觸發 | 分析、滾動分析都要使用者按按鈕。系統不會自己偷跑 |
| 對話一天一段 | 問 AI 的歷史用台灣日期切，隔天自動歸零 |

---

## 二、資料流：單向，沒有回頭箭頭

**寫入（只有按「分析」會觸發）**

```
日記原文  entries
   ↓ 按「分析」，一次 LLM
事實      facts（一條一句、有分類、有向量）
   ↓ 按「更新」，另一次 LLM
滾動摘要  profiles（我是怎樣的人 / pattern / 能量建議）
```

**讀取（問 AI 時）**

```
問題 → 向量相似度 + 分類篩選 → 撈到的 facts
     → facts + profile + 問題 → LLM → 回答（標「出自 3/12」）
```

**問答路徑沒有寫入能力**：不產生 entry、不產生 fact。程式碼裡就不存在那個呼叫。
使用者亂聊不會污染記憶 —— 不是「不會」，是**做不到**。

---

## 三、資料模型

### `entries`（改）— 一天一篇日記

| 欄位 | 說明 |
|---|---|
| `user_id` + `entry_date` | **unique 索引**。`entry_date` 是台灣日期（`clock.py`） |
| `content` | 本人寫的日記全文 |
| `energy` | 自評 1–10 的整數，寫完日記才拉 |
| `analysis_count` | 只算分析次數，上限 3；寫日記不計 |
| `analyzed_at` | 有值代表分析過 |

**移除**：`transcript` / `ai_reply`（合併進 `content`）、`mood` / `themes` 廢欄位。

### `facts`（幾乎不動）— 分類加一個

```python
CATEGORIES = (
    "about me", "preferences", "people", "work & career",
    "goals & aspirations", "health & habits", "beliefs",
    "patterns", "wins", "gratitude",     # ← 新增 gratitude
)
```

**能量不進 facts** —— 它是數字，不是可以單獨讀懂的一句話。放進來只會得到「他今天能量 70%」這種沒辦法搜尋也沒辦法比較的廢事實，還會稀釋檢索。

### `profiles`（改結構）— 滾動摘要

`content` 從一坨純文字改成三段：**我是怎樣的人**（強調勝利）、**我的 pattern**、**如何提升能量**。
`entry_count` 拿來顯示「上次更新是 N 篇日記以前」。

### `questions`（新）— 問答紀錄

| 欄位 | 說明 |
|---|---|
| `user_id`, `asked_date` | 台灣日期，用來切「今天這段對話」 |
| `question`, `answer` | 問與答 |
| `sources` | 引用到哪幾天（顯示「出自 3/12」用） |

跟 `entries` 完全隔離。

## 四、畫面：4 個 tab（手機優先）

### 1. 紀錄（首頁）

由上而下：

1. **能量長條圖** — 7 天 / 30 天兩檔切換
2. **今天的日記卡** — 寫／編輯／拉能量／按分析（顯示剩幾次）
3. **每天一張小卡**，範圍跟著上面的圖表走，沒寫的日子不出現

```
7/22  ●70%   🏆 談成了那個案子
             🙏 阿明幫我看了履歷        +2
```

只顯示第一條勝利、第一條感恩，其餘收成 `+2`。點進去看當天全文。
往下滑時顏色點連成一條，跟上面的圖互相呼應。

**能量顏色帶**（輸入 1–10，顯示 ×10 成 %）：

| 分數 | 顯示 | 顏色 |
|---|---|---|
| 1–3 | 10–30% | 🔴 紅 |
| 4–6 | 40–60% | 🟡 黃 |
| 7–10 | 70–100% | 🟢 綠 |

輸入用 10 段而不是 0–100，因為沒有人分得出來自己今天是 67 還是 71 —— 假精確會讓資料變髒、趨勢圖失去意義。

**圖表套件：Recharts**。每根 bar 各自上色 + 標數字 + 檔位切換都是原生支援，不手刻。

### 2. 分析

滾動摘要三段 + 一顆「更新」按鈕 + 「上次更新是 N 篇日記以前」。

### 3. 問 AI

聊天介面，**保留串流打字機與 TTS**（技術亮點，質感差很多）。
答案標出自哪幾天。左側／上方有按日期分組的問答紀錄列表。

### 4. Mantra

自己喜歡的 affirmation 收藏，沿用現有功能。

## 五、刪除清單

沒用到的直接刪檔，不要留著。

- **TTS 只留在問 AI**；日記那條路完全不碰 `voice` 的朗讀
- **日記分析不串流** —— 一次出完，給轉圈就好
- 每輪對話寫 entry / 抽 fact 的呼叫（`agent.py` 裡的 journal 寫入）
- 舊的 `coach` / `journal` route 中沒有對應新頁面的 endpoint
- `entries` 的 `mood` / `themes` 欄位（migration drop）
- 前端 App.jsx 中屬於舊單頁流程的部分（重寫成 4 tab）

## 六、舊資料遷移

**備份**：`~/Documents/minerva-backups/minerva-2026-07-22.dump`（pg_dump 自訂格式，repo 外）。

線上實際是 **62 筆 entries，只橫跨 7 天**：

```
7/15  32 筆   ← 開發測試灌的，丟掉
7/16  14 筆   ← 同上，丟掉
7/17   1 筆
7/18   2 筆
7/19   3 筆   ← 這 5 天是真的，合併保留
7/20   6 筆
7/21   4 筆
```

**做法：只留 7/17–7/21 五天。** 同一天的 `transcript` 接起來成一篇 `content`，`ai_reply` 丟掉。
7/15、7/16 連同它們的 facts 與 embedding 一起刪。

合併後手動幫這 5 天各補一個 `energy` 分數。

**facts 全部重跑**：舊的 424 條是從「對話輪」抽的，`entry_id` 也對不上合併後的新 entry。
清空 `facts` 與 `langchain_pg_embedding`，用新的 10 分類（含 `gratitude`）對 5 篇日記重新抽一次。
`scripts/backfill_facts.py` 已經有這件事的雛形，改成吃新的 entry 結構即可。
